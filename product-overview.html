ML + Product Logic

Cross-Sell Timing Analyzer

A sentiment-driven decision engine that tells e-commerce chatbots when to cross-sell — and when to hold back.

March 2026 · Built by Sandhya Godavarthy

▶ Live Demo 💻 GitHub

Opportunity Statement

E-commerce chatbots today recommend products without factoring in the customer's emotional state during a support conversation. An agent resolving a frustrated customer's delayed delivery might trigger a cross-sell suggestion — missing the fact that the customer isn't in the right mindset. Conversely, when a customer is genuinely happy after a smooth resolution, the agent misses the optimal window to suggest complementary products.

The question this project answers: "Given a live customer conversation, is this the right moment to cross-sell, or will it backfire?"

How It Works

The system uses a two-layer approach: an ML model for sentiment scoring and a product logic layer for business decisions.

Customer message
→
ML Model scores sentiment
→
Product logic decides
→
✅ Cross-sell  /  ⏳ Wait  /  🛑 Hold back

Layer 1: Sentiment Scoring

Each customer message is scored from -1.0 (very negative) to +1.0 (very positive) using Twitter RoBERTa, a Transformer model pre-trained on 124 million tweets and fine-tuned for 3-class sentiment classification (positive / neutral / negative). Runs locally via HuggingFace — no API key, ~50ms per message.

Layer 2: Cross-Sell Decision

The sentiment score feeds into a threshold-based decision engine:

🟢
Green Zone
Score ≥ 0.3
Cross-sell now
🟡
Yellow Zone
-0.2 to 0.3
Wait
🔴
Red Zone
Score ≤ -0.2
Hold back

Thresholds are configurable in the UI — different brands and contexts need different sensitivity.

Key Product Decisions

1

Chose 3-class Twitter RoBERTa over 2-class DistilBERT

The initial version used DistilBERT (movie reviews, positive/negative only). Critical flaw: no neutral class. A casual "what are you up to?" scored 0.98 positive — triggering a false cross-sell. Twitter RoBERTa adds neutral detection, handles casual language, and was trained on 124M tweets matching the tone of support chat.

2

Per-message scoring is MVP — not production-ready

Current implementation scores each message independently. Limitation: a customer furious 2 minutes ago who says "okay thanks" gets scored positive. That's grudging acceptance, not genuine satisfaction.

Production-ready approach: trigger on "resolved positive" — the pattern where (1) customer had a problem, (2) agent resolved it, (3) sentiment genuinely shifted positive over multiple messages. This requires conversation-level context via rolling weighted averages, not per-message scoring.

3

Configurable thresholds over hardcoded rules

A luxury hotel brand may want a higher threshold than a grocery service. The UI exposes sliders so product teams tune sensitivity per brand without code changes.

4

Cascade architecture for production scale — all ML, no LLM dependency

The production pipeline uses trained ML models at every tier — fast, cheap, and scalable. No LLM required in steady state.

Tier 1
Every message: Per-message sentiment model (Twitter RoBERTa, ~50ms, free) handles obvious green/red cases — ~80% of conversations
Tier 2
Ambiguous cases: Conversation-level ML classifier (XGBoost or fine-tuned Transformer) trained on labelled conversations — handles the remaining ~20% using features like sentiment trajectory, resolution patterns, and conversation length
Tier 3
Batch, post-conversation: Fine-tuned classifier auto-labels cross-sell attempts and outcomes for the analytics dashboard

Where does an LLM fit? Only as a bootstrapping tool. When you have zero labelled data, an LLM can generate the first batch of training labels — "read this conversation and tell me if a cross-sell was attempted and whether it succeeded." Once you've collected 5-10K labelled conversations this way, you train your own classifier and the LLM is no longer needed. The LLM is a stopgap, not a production dependency.

5

Labelled outcomes validate the approach

The dataset includes outcome labels (cross_sell_success / cross_sell_failure). The dashboard proves that cross-selling in positive sentiment windows converts at significantly higher rates than pitches during unresolved complaints.

Improvements for Production

1. Conversation-level sentiment

Replace per-message scoring with rolling weighted averages. Ideal trigger: resolved positive — detect the pattern where sentiment went negative → agent intervened → sentiment recovered over 2-3 messages.

2. Auto-detection of cross-sell attempts

Replace manual labelling with keyword rules or an LLM classifier to automatically detect when an agent pitched a product and whether it was accepted.

3. LLM for crafting the cross-sell pitch message

Classification and scoring don't need an LLM — but crafting the actual cross-sell message does. Once the system decides it's the right moment to pitch, an LLM can generate a personalized recommendation based on conversation context: "Since you just resolved your TV delivery, here's a wall mount that fits your model." This is where LLM creativity genuinely adds value — not in deciding when to sell, but in deciding what to say.

4. Domain-specific fine-tuning

Fine-tune Twitter RoBERTa on labelled customer support conversations from the specific domain for improved accuracy.
